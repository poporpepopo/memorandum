# システム仕様書：Whisper & Gemma 4 搭載・会議リアルタイム要約ツール「memorandum」

## 1. 製品概要

本システムは、ローカルの AI 演算リソースのみを利用し、外部クラウドに依存することなく会議の音声認識と要約を行うパーソナル議事録アシスタントである。
Web 会議（再生音声のループバック録音）と対面会議（マイク録音）の両方に対応し、「1 分間」という一定のリズムで要約をデスクトップ通知（Push 型）およびターミナル出力（Pull 型）の両面で提供する。

対応プラットフォーム: **Windows 10/11 · macOS**（単一のソースコードで両対応）

## 2. システムアーキテクチャ

データプライバシーの確保と低レイテンシを実現するため、すべての処理をローカル環境で完結させる「エッジ AI」構成をとる。
録音（プロデューサー）と AI 処理（コンシューマー）を別スレッドに分離し、`queue.Queue` で接続することで、推論に時間がかかっても録音が途切れない設計とした。

### 2.1 構成コンポーネント

| クラス | 役割 |
| --- | --- |
| `AudioSource` (Protocol) | 音源の共通インターフェース。以下 2 実装を透過的に切替 |
| `SystemAudioCapture` | **Web 会議向け。** Windows の WASAPI ループバックでスピーカー / イヤホンへの再生音声を録音（PyAudioWPatch）。デバイスレートからの 16kHz リサンプリングとモノラル化を内包 |
| `MicrophoneCapture` | **対面向け。** SpeechRecognition / PyAudio によるマイク録音（16kHz） |
| `Transcriber` | OpenAI Whisper (small) によるローカル文字起こし。無音確信度・定型文によるハルシネーション除去を内包 |
| `Summarizer` | Ollama (Gemma 4 E4B) によるチャンク要約・最終要約の生成 |
| `Notifier` | デスクトップ通知（Windows: winotify トースト / macOS: plyer）。失敗しても本体処理を停止させない |
| `MeetingAssistant` | オーケストレーター。録音スレッドと処理スレッドの制御、終了処理、議事録保存 |

### 2.2 処理パイプライン

```
[録音スレッド]  AudioSource → RMS 無音フィルタ → queue.Queue
[処理スレッド]  queue.Queue → Whisper 文字起こし → ハルシネーションフィルタ
                → Gemma 4 要約 → デスクトップ通知 / ターミナル表示
[終了時]        残キュー処理 → 全原文の最終要約 → 議事録ファイル保存
```

音声はディスクを経由せず、`NumPy float32` 配列としてメモリ上で直接 Whisper へ渡される。

## 3. 主要機能仕様

### 3.1 音源選択（`--source`）

* **system（Windows 既定）:** 既定の再生デバイスに出力される音声を WASAPI ループバックで録音。Web 会議の相手音声を仮想デバイスなしで取得できる。
* **mic（macOS 既定）:** マイク入力。起動時に環境ノイズを 1 秒計測して感度を自動調整する。

### 3.2 リアルタイム・スライディング要約

* **動作周期:** 60 秒（`--chunk-seconds` で変更可能）
* **内容:** 直近 1 分間の発言から重要事項を抽出し、30 文字程度のプレーンテキストで通知。
* **並行性:** 文字起こし・要約の処理中も次のチャンクの録音を継続（発言の取りこぼしなし）。

### 3.3 異常動作（ハルシネーション）抑制

三段階のフィルタリングで、無音時における Whisper 特有のループ出力を抑制する。

1. **物理基準:** RMS（Root Mean Square）音圧がしきい値（既定 0.005）未満のチャンクは推論せず破棄
2. **モデル基準:** Whisper の `no_speech_threshold=0.6` / `logprob_threshold=-1.0` で低信頼セグメントを破棄
3. **テキスト基準:** 既知の定型ハルシネーション文（「ご視聴ありがとうございました」等）と 5 文字未満の断片を破棄

### 3.4 ハイブリッド・ログ管理

* **標準出力:** ターミナル上に「原文」（タイムスタンプ付き）と「要約」を対で出力。
* **永続化保存:** `Ctrl+C`（`KeyboardInterrupt`）検知時に未処理キューを処理し切り、全原文から「全体俯瞰要約」を生成して `meeting_log_YYYYMMDD_HHMMSS.txt`（UTF-8）へ保存。最終要約の生成に失敗した場合も原文は必ず保存する。

### 3.5 起動時セルフチェック（fail fast）

* Ollama サーバーへの接続と指定モデルの存在を起動直後に検証し、問題があれば対処方法を明示してただちに終了する。

## 4. CLI インターフェース

| オプション | 既定値 | 説明 |
| --- | --- | --- |
| `--source {system,mic}` | OS により自動 | 録音する音源 |
| `--llm-model` | `gemma4:e4b` | 要約に使う Ollama モデル |
| `--whisper-model` | `small` | Whisper モデルサイズ |
| `--language` | `ja` | 文字起こし言語 |
| `--chunk-seconds` | `60` | 録音サイクル秒数 |
| `--rms-threshold` | `0.005` | 無音判定の音圧しきい値 |
| `--output-dir` | カレント | 議事録保存先 |
| `--no-notify` | — | 通知の無効化 |

## 5. 動作環境・依存関係

* **OS:** Windows 10/11 または macOS（Apple Silicon 推奨）
* **ソフトウェア依存:**
  * Python 3.9+
  * Ollama（Gemma 4 E4B モデル。メモリ 8GB クラスは E2B を推奨）
  * 依存ライブラリは `requirements.txt` に集約（Windows 専用依存は環境マーカーで自動切替）
* **備考:** 音声はメモリ上の配列として直接 Whisper へ渡すため、FFmpeg は不要。

## 6. セキュリティ仕様

* **非通信ポリシー:** 音声データおよびテキストデータは、外部サーバー（Google, OpenAI, クラウドプロバイダ等）へ一切送信されない。Ollama との通信も `localhost` に限定される。
* **機密保持:** すべての計算プロセスは揮発性メモリおよびローカルストレージ内で完結する。
