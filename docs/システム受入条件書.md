# システム受入条件書 (Acceptance Criteria)

検証状況の凡例 — ✅: 実機検証済み (2026-07-11, Windows 11) / ⬜: 未検証

## 1. 音声入力および信号処理層

* **[AC-1.1] 音源の選択** ✅
  * `--source system` で既定の再生デバイスへの出力音声（Web 会議の相手音声）が WASAPI ループバックにより録音されること。
  * `--source mic` でマイク入力が録音されること。
  * 既定値が OS に応じて自動選択されること（Windows: system / macOS: mic）。
* **[AC-1.2] 音声キャプチャの安定性** ✅（起動〜録音ループ到達まで確認）
  * 1 分間の連続録音において、途中でプロセスがクラッシュせず、安定して音声バッファが生成されること。
* **[AC-1.3] サンプリング変換** ✅
  * ループバック録音時、デバイスのサンプルレート（例: 48kHz ステレオ）から Whisper 用の 16kHz モノラルへ正しく変換されること。
* **[AC-1.4] ハルシネーション（幻聴）の抑制** ✅（無音音声で確認）
  * **物理基準:** RMS 値がしきい値（既定 0.005）未満の無音チャンクで、Whisper への推論リクエストがスキップされること。
  * **モデル基準:** `no_speech_threshold` / `logprob_threshold` により無音時のループテキストが出力されないこと。
  * **テキスト基準:** 既知の定型文（「ご視聴ありがとうございました」等）と 5 文字未満の断片が破棄されること。
* **[AC-1.5] 録音デバイス喪失からの自動復旧** ⬜（実装済み・切断試験は未実施）
  * ループバック録音中に既定の再生デバイスが無効化された場合（モニタのスリープ、再生デバイスの切替等）、最新の既定デバイスを取り直して自動的に録音を再開すること。
  * エラーが連続する場合（5 回）は、原因を明示したメッセージで終了へ移行し、それまでの議事録が保存されること。

## 2. 音声認識 (STT) および要約 (LLM) 層

* **[AC-2.1] ローカル認識精度** ✅（合成音声によるループバック E2E テストで一致を確認）
  * Whisper (small) モデルにより、日本語の会話内容が文脈を維持したままテキスト化されること。
* **[AC-2.2] 要約の適切性** ✅
  * Gemma 4 E4B により、直近 1 分の発言が 30 文字程度のプレーンテキストで要約されること。
* **[AC-2.3] 出力フォーマットの遵守** ✅
  * 要約結果に Markdown 記号（`#`, `*`, `` ` ``）が含まれず、通知バナーに最適化された形式であること。
* **[AC-2.4] 起動時セルフチェック** ✅
  * Ollama サーバー未起動・モデル未取得の場合、起動直後に対処方法を明示したエラーで終了すること（fail fast）。

## 3. ユーザーインターフェースおよび通知層

* **[AC-3.1] 通知の周期性と同期** ✅（winotify によるトースト送信を確認）
  * 録音終了後、遅滞なく（目標 10 秒以内）デスクトップ通知が表示されること。
  * Windows では winotify（トースト API）、macOS では plyer を使用すること。
* **[AC-3.2] リアルタイム・モニタリング** ✅
  * ターミナル（stdout）上に、処理中の「原文」（タイムスタンプ付き）とそれに対する「要約」が対になって表示されること。
* **[AC-3.3] 通知失敗の隔離** ⬜
  * デスクトップ通知が失敗しても、録音・文字起こし・要約・保存の本体処理が継続すること。
* **[AC-3.4] ワンクリック起動** ✅（Windows のみ。macOS は構文検証まで）
  * `run.bat`（Windows）/ `run.command`（macOS）のダブルクリックで、初回セットアップ（venv 作成・依存インストール）から起動まで完了すること。
  * Windows では Ollama サーバーが未起動の場合、`run.bat` が自動起動して接続を待機すること（Ollama 停止状態からの起動を実機確認済み）。
* **[AC-3.5] コンソールエンコーディング耐性** ✅
  * Windows で stdout をリダイレクトした場合（cp932 環境）でも、絵文字・日本語出力で `UnicodeEncodeError` が発生しないこと。

## 4. データ永続化および終了処理

* **[AC-4.1] 終了シグナルの正常処理** ⬜
  * `Ctrl+C` 入力時に即座に強制終了せず、未処理キューの消化 → 最終要約の生成 → 保存へ移行すること。
* **[AC-4.2] 最終レポートの完全性** ⬜
  * 会議全体の原文を結合し、それに基づいた「全体俯瞰要約」が生成されること。
  * 最終要約の生成に失敗した場合でも、原文の議事録は保存されること。
* **[AC-4.3] ファイル出力の正確性** ✅
  * `meeting_log_YYYYMMDD_HHMMSS.txt` 形式のファイルが生成され、最終要約とタイムスタンプ付き全原文が UTF-8 エンコーディングで保存されていること。
  * `--output-dir` で保存先を変更でき、ディレクトリが無ければ作成されること。

## 5. 非機能要件（リソース・セキュリティ・並行性）

* **[AC-5.1] オフライン動作** ⬜
  * インターネット接続を遮断した状態（Air-gapped）で、すべての機能が正常に動作すること（Ollama との通信は localhost のみ）。
* **[AC-5.2] 負荷耐性** ⬜
  * 1 時間以上の連続稼働において、メモリ使用量が著しく増大（メモリリーク）せず、パフォーマンスが維持されること。
* **[AC-5.3] 録音の連続性** ⬜（設計上保証・長時間実測は未実施）
  * 文字起こし・要約の処理中も録音が継続し、発言が取りこぼされないこと。
